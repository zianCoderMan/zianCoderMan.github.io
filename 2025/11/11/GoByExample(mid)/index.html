9.通道的关闭

注：为什么把 close(jobs) 注释掉后，会运行时死锁（deadlock）

核心原因：子协程永远不知道 “任务已经发完了”

要理解这个问题，必须回到「发送方和接收方的协作逻辑」：

1. 注释 close(jobs) 后，各方的状态

  • 主协程：发送完 3 个任务（1、2、3）后，打印 sent all jobs，然后阻塞在 <-done，等待子协程发送 “处理完毕” 的信号；
  • 子协程：在for循环中持续执行j, more := <-jobs：
    • 首先接收缓冲区中的 3 个任务（1、2、3），more 始终为 true（因为通道没关闭）；
    • 处理完这 3 个任务后，缓冲区空了，但通道仍处于「未关闭状态」；
    • 子协程会继续阻塞在 <-jobs，等待新的任务（它不知道主协程已经不发了）；
  • 关键矛盾：主协程在等 done 信号，子协程在等新任务，双方互相等待，永远无法推进 —— 这就是死锁。

2. close(jobs) 的核心作用：打破 “互相等待”

close(jobs) 不是 “可选操作”，而是「发送方向接收方传递 “任务终止” 的唯一信号」：

  • 关闭通道后，子协程处理完缓冲区的 3 个任务，再执行 j, more := <-jobs 时，会得到 more = false（通道关闭 + 缓冲区空）；
  • 此时子协程才知道 “没有更多任务了”，会执行 done <- true，通知主协程退出；
  • 主协程收到 done 信号后，解除阻塞，程序正常结束。

11.Timer

Timer 的核心基础

1. Timer 的本质

Timer 是一个 “单次定时任务器”，核心逻辑：

  • 调用 time.NewTimer(d) 时，启动一个后台协程；
  • 延迟 d 时间后，后台协程向 Timer.C（一个 chan time.Time 通道）发送当前时间值；
  • 使用者通过「接收 Timer.C 的信号」，感知定时时间到达；
  • 核心：Timer 触发一次后就失效（单次触发），如需重复定时需重新创建或重置。

2. 核心属性与方法

属性 / 方法 作用说明
Timer.C 只读通道（<-chan time.Time），定时时间到后会收到一个时间信号（信号本身可忽略）；
time.NewTimer(d) 创建 Timer 实例，延迟 d 后触发；
timer.Stop() 提前停止 Timer：返回 true 表示 Timer 未触发且成功停止，false 表示已触发或已停止；
timer.Reset(d) 重置 Timer：停止当前未触发的 Timer，重新设置延迟 d（可实现重复定时）；

3. 与 time.After 的区别（关键）

  • time.After(d)：是 Timer 的简化版，返回 Timer.C 通道，无法提前停止（Timer 会在触发后自动回收）；
  • Timer：支持 Stop() 提前停止，更灵活（适合需要取消定时的场景）；
  • 核心差异：Timer 可控制（停止 / 重置），time.After 不可控制。

注：关键特性

1. Timer 触发后，Timer.C 通道不会自动关闭

Timer 触发一次后，Timer.C 通道会保留发送的时间值（缓冲区容量 1），再次接收会立即返回该值，不会阻塞：

  • 运行结果：第二次接收成功（无阻塞）；
  • 原因：Timer.C 是容量为 1 的缓冲通道，触发后数据会保留在通道中。

2. 提前停止 Timer 后，Timer.C 通道会永久阻塞

如案例 2 所示，timer.Stop() 后，后台协程不会向 Timer.C 发送信号，通道会一直处于 “无数据” 状态，接收操作会永久阻塞 —— 若子协程中只有 <-timer.C 而无其他逻辑，会导致子协程泄漏（一直占用资源）。

12.Ticker

Ticker 的核心基础

1. Ticker 的本质

Ticker 是 “周期性定时任务器”，核心逻辑：

  • 调用 time.NewTicker(d) 时，启动一个后台协程；
  • 每隔 d 时间，后台协程向 Ticker.C（chan time.Time 通道）发送当前时间值；
  • 使用者通过接收 Ticker.C 的信号，触发周期性任务；
  • 核心：只要不停止 Ticker，就会一直周期性触发（直到程序退出或主动停止）。

ticker.Stop() 停止 Ticker：关闭后台协程，不再向 Ticker.C 发送信号（必须主动停止，否则后台协程会泄漏）；

13.工作池

工作池的核心基础

工作池的本质是「生产者 - 消费者模型」的具体化，三个核心组件缺一不可：

  1. 任务生产者：负责生成任务，发送到「任务通道（Jobs）」（代码中主协程是生产者）；
  2. 工作协程（Worker）：提前创建的固定数量协程，从任务通道获取任务执行，执行结果发送到「结果通道（Results）」（代码中 worker 函数是 Worker）；
  3. 任务 / 结果通道：
    • 任务通道（jobs <-chan int）：单向接收通道，Worker 只能从这里拿任务；
    • 结果通道（results chan<- int）：单向发送通道，Worker 只能往这里存结果；
    • 通道的缓冲容量：通常设置为任务总数（避免生产者发送任务时阻塞）。

核心逻辑：生产者往 Jobs 发任务 → 多个 Worker 并发抢任务执行 → 执行结果存入 Results → 生产者等待所有结果处理完毕。

关键特性

1. 工作池的核心优势：并发控制 + 资源复用

  • 并发控制：通过固定 Worker 数量（3 个），限制最大并发数，避免系统因协程过多导致 CPU / 内存过载；
  • 资源复用：Worker 是提前创建的，处理完一个任务后会继续处理下一个，无需频繁创建销毁协程（创建协程有资源开销，频繁创建会降低效率）。

for range jobs 会自动处理「任务接收 + 通道关闭检测」：

  • 有任务时，接收任务并执行；
  • 通道关闭且无任务时，自动退出循环，Worker 协程自然结束（无需手动判断 ok 值）；

14.WaitGroup

WaitGroup 是 Go 中最常用的协程同步工具之一（尤其适合 “无返回结果的批量任务”，如日志打印、资源清理），比之前用「通道（done chan bool）」同步更简洁（无需手动创建通道、发送信号）。

WaitGroup 的核心基础

WaitGroup 本质是一个 “计数器”，通过三个方法协同工作，实现协程同步：

方法名 作用说明
wg.Add(n int) 增加等待的协程数量（n 是要新增的协程数），必须在启动协程前调用；
wg.Done() 减少等待的协程数量（等价于 wg.Add(-1)），表示 “当前协程执行完毕”，必须在协程退出前调用；
wg.Wait() 主协程阻塞，直到等待的协程数量变为 0（所有子协程都调用了 Done()），才会解除阻塞；

核心逻辑：

  1. 主协程用 wg.Add(n) 声明 “要等待 n 个协程”；
  2. 每个子协程启动后，用 defer wg.Done() 确保 “执行完毕后自动通知 WaitGroup”；
  3. 主协程用 wg.Wait() 阻塞，直到所有子协程都通知 “完成”，再继续执行。

关键特性

1. 必须避免的 3 个常见错误

（1）wg.Add(n) 调用时机错误（在协程内调用）

  • 后果：主协程可能在所有 wg.Add(1) 执行前就调用 wg.Wait()，此时计数为 0，主协程直接退出，子协程被终止；
  • 原则：wg.Add(n) 必须在 启动协程前 调用，确保主协程 wg.Wait() 时，计数已经正确设置。

（2）忘记调用 wg.Done() 或 defer 位置错误

  • 后果：主协程会永久阻塞在 wg.Wait()（计数永远不为 0）；
  • 原则：wg.Done() 必须用 defer 声明，且放在 协程函数开头（确保任务执行前就注册 “完成通知”，无论任务是否 panic 都能触发）。

（3）拷贝 WaitGroup 变量（值类型传递）

  • 原因：sync.WaitGroup 是值类型，拷贝后会生成新的实例，子协程操作的是拷贝后的实例，原主协程的 wg 计数不会变化；

速率限制的核心原理

速率限制的本质是「控制 “任务执行” 与 “时间” 的匹配关系」，Go 实现的核心逻辑：

  1. 周期性信号源：用 time.Tick(d) 生成周期性信号（如每 200ms 一个信号），time.Tick(d) 本质是一个 <-chan time.Time 通道，每隔 d 时间发送一次当前时间；
  2. 信号与任务绑定：每个任务执行前，必须先接收一个周期性信号 —— 通过通道的阻塞特性，强制任务执行频率不超过信号生成频率（如每 200ms 只能接收一个信号，所以每秒最多执行 5 个任务）；
  3. 突发流量支持：用缓冲通道存储 “预分配的信号”，允许短时间内处理多个突发任务（缓冲容量 = 最大突发数），后续任务仍按固定速率执行。

核心思想：用 “信号发放速率” 控制 “任务执行速率”，通道的阻塞特性天然实现了 “任务等待信号” 的逻辑，无需手动计时或加锁。

16.原子计数器

原子操作的底层是通过 CPU 硬件指令 实现的（如 CAS 指令），比互斥锁（Mutex）更高效（无锁竞争、上下文切换开销）。

原子计数器的核心基础

1.核心问题：为什么需要原子计数器？

多协程并发修改同一个变量时，会出现「数据竞争」—— 普通变量的增减操作（如 ops++）不是原子的，本质是 “读取→修改→写入” 三步，多协程同时操作可能导致数据错乱。

示例（普通变量的并发问题）：

  • 问题：ops++ 被拆解为「读 ops 值 → 加 1 → 写回 ops」，多个协程可能同时读同一个值，导致 “加 1 操作被覆盖”（如协程 A 和 B 都读 ops=100，加 1 后都写回 101，实际应是 102）；
  • 解决：用 sync/atomic 包的原子操作，将 “读 - 改 - 写” 封装为一个不可中断的硬件级原子指令，避免数据竞争。

2. sync/atomic 核心原子计数方法（针对无符号整数）

方法名 作用说明
atomic.AddUint64(&x, n) 对无符号 64 位整数 x 原子加 n，返回加后的新值（x 必须传指针）；
atomic.LoadUint64(&x) 原子读取 x 的值（避免读取时被其他协程的写操作干扰）；
atomic.StoreUint64(&x, v) 原子将 v 赋值给 x（避免赋值时被其他协程的读 / 写操作干扰）；
atomic.CompareAndSwapUint64(&x, old, new) CAS 操作：若 x 等于 old，则原子替换为 new，返回是否成功（乐观锁核心）；
  • 注意：sync/atomic 针对不同类型提供了对应的方法（如 AddInt32/AddInt64 对应有符号整数），需根据变量类型选择，不可混用。

3. 原子操作的核心优势（对比 sync.Mutex）

特性 原子操作（atomic） 互斥锁（sync.Mutex）

1. 必须避免的 3 个常见错误

（1）原子操作变量未传指针

  • 原因：原子操作需要直接修改变量的内存地址，必须传指针（&ops），否则无法保证操作的原子性；
  • 原则：所有 atomic 包的方法，第一个参数都是变量的指针（如 *uint64、*int32）。

（2）混用原子操作和普通操作

  • 后果：普通操作（ops++）非原子，会干扰原子操作的安全性，导致数据错乱；
  • 原则：一个变量若用 atomic 包操作，就必须全程用原子操作（增、删、读、写），不能混用普通操作。

（3）原子操作用于复杂逻辑

  • 问题：atomic.AddUint64 是原子的，但 “加 1 + 判断” 整体不是原子的 —— 可能多个协程同时加 1 后，都判断超过阈值，导致重复执行后续逻辑；

2. 原子操作的适用场景（vs 互斥锁）

  • 优先用原子操作的场景：
    1. 简单数值计数（如接口调用次数、任务执行量、错误统计）；
    2. 高频次操作（如每秒上万次的计数，原子操作性能优势明显）；
    3. 单一变量的读 / 写 / 增减（原子操作仅支持单一变量）。
  • 优先用互斥锁的场景：
    1. 复杂逻辑（如多步操作、多变量修改）；

18.状态协程

状态协程的核心优势是「天然并发安全，无需手动加锁」—— 所有对共享状态的操作都集中在一个协程中，不存在多协程并发修改的问题，通道的 FIFO 特性还能保证请求处理的顺序性。

核心原理：状态协程的 “单线程处理” 思想

传统多协程操作共享状态（如 map），需要用锁（sync.Mutex）保护临界区；而状态协程的思路是：

  1. 状态隔离：共享状态（state map[int]int）只由一个 “状态协程” 持有，其他协程无法直接访问；
  2. 通道通信：读写请求通过专用通道（reads chan readOp 读请求、writes chan writeOp 写请求）发送给状态协程；
  3. 串行处理：状态协程通过 select 监听读写通道，收到请求后串行处理（同一时间只处理一个请求），天然避免数据竞争。

简单说：用 “协程串行处理” 替代 “多协程并发竞争 + 锁”，通道负责请求传递和同步，状态协程负责逻辑执行，既安全又简洁。

补充1：通道为什么天生线程安全？

Go 的通道在底层由运行时（runtime）管理，内置了同步机制，核心保障有两点：

  1. 发送 / 接收操作是原子的：
    • 发送操作（ch <- data）：将数据存入通道缓冲区（或直接传递给接收方）的整个过程，不会被其他协程打断；
    • 接收操作（data := <-ch）：从通道缓冲区取出数据（或直接接收发送方数据）的整个过程，也不会被其他协程打断；
    • 原子性意味着：不会出现 “多个协程同时写通道导致数据错乱”“多个协程同时读通道导致重复读取” 的问题。
  2. 运行时的锁机制：
    • 通道底层维护了一个互斥锁（mutex），每次发送 / 接收操作都会先获取锁，操作完成后释放锁；
    • 锁机制确保：同一时间只有一个协程能执行通道的发送或接收操作，天然避免了数据竞争。

需要注意的 “非线程安全” 场景（不是通道本身的问题）

通道本身是线程安全的，但以下场景可能出现问题，需要避免 —— 本质是 “用户代码逻辑问题”，而非通道的安全问题：

  1. 向已关闭的通道发送数据：会触发 panic（通道关闭后禁止发送，这是语法规则，不是线程安全问题）；
  2. 重复关闭通道：会触发 panic（同样是语法规则）；
  3. 通道的 “外层数据” 共享：如果多个协程通过通道传递的是 “指针 / 引用类型”（如切片、map），且同时修改该数据的内容，会出现数据竞争 —— 但这是 “共享数据的线程安全问题”，和通道本身无关。

Go 通道 vs Java 多线程

Go 的通道（chan）和 Java 的多线程模型（核心是 Thread + synchronized/Lock + 共享内存），本质是 两种不同的并发编程范式：

  • Go 通道是「通过通信共享内存（CSP 模型）」的核心载体，强调 “协程间通过通道传递数据，而非直接共享内存”；
  • Java 多线程是「通过共享内存通信」的经典模型，强调 “多线程竞争共享资源，需手动加锁同步”。
维度 Go 通道（基于 CSP 模型） Java 多线程（基于共享内存模型）

补充2：CAS 为什么能避免锁竞争的开销

要搞懂 “CAS 为什么能避免锁竞争的开销”，核心是先明确 “锁竞争的开销到底是什么”，再对比 CAS 的 “无锁机制” 如何绕开这些开销 —— 本质是「锁用 “阻塞等待” 解决并发，CAS 用 “循环重试” 解决并发」，后者省去了阻塞 / 唤醒的核心开销。

一、先搞懂：锁竞争的开销到底来自哪里？

我们以之前提到的 sync.Mutex（互斥锁）为例，当多个协程竞争同一个锁时，开销主要集中在 3 个核心环节（这些都是 CAS 没有的）：

1. 内核态与用户态切换（最大开销）

  • 互斥锁的实现依赖 操作系统内核：当协程 A 拿到锁后，其他协程（B、C、D）尝试加锁会失败；
  • 失败的协程不会一直占用 CPU 重试，而是会被操作系统 挂起（从 “运行态” 转为 “阻塞态”），这个过程需要从 “用户态” 切换到 “内核态”（操作系统负责调度）；
  • 当协程 A 释放锁后，操作系统还要从 “内核态” 切换回 “用户态”，唤醒被挂起的协程（B、C、D 中的一个），让它拿到锁继续执行；
  • 这种 “用户态↔内核态” 切换的开销极大（单次切换约几十到几百纳秒），远超 CAS 操作本身（几纳秒）。

2. 协程 / 线程上下文切换

  • 被挂起的协程，其当前的执行状态（如 CPU 寄存器值、程序计数器）需要被操作系统保存到内存中；
  • 唤醒时，又要从内存中恢复这些状态到 CPU 中 —— 这个 “保存→恢复” 的过程就是 “上下文切换”，同样有显著开销（尤其当竞争激烈、切换频繁时）。

3. 锁的调度与排队开销

  • 操作系统需要维护一个 “等待队列”，管理所有因拿不到锁而阻塞的协程；
  • 释放锁时，还要从队列中选择一个协程唤醒（公平锁 / 非公平锁的调度逻辑），这个过程也需要消耗内核资源。

简单说：锁竞争的核心开销，是 “阻塞→内核调度→唤醒→上下文切换” 这一整套流程的开销—— 本质是用 “让协程等” 的方式解决竞争，而 “等” 的过程本身就很耗资源。

二、CAS 为什么能避免这些开销？核心是 “无锁 + 用户态重试”

CAS（Compare And Swap）的核心是「不阻塞协程，仅在用户态循环重试」，完全绕开了上述锁竞争的开销，具体逻辑如下：

1. 全程在用户态执行，无内核态切换

  • CAS 是 CPU 硬件支持的原子指令（如 x86 架构的 CMPXCHG 指令），整个 “比较 + 交换” 操作都在 用户态 完成，不需要调用操作系统内核；
  • 即使 CAS 失败（遇到并发竞争），也不会挂起协程，只是在用户态重新执行 “读旧值→算新值→CAS” 的循环 —— 全程不涉及内核态切换，开销极低。

2. 无协程阻塞 / 唤醒，无上下文切换

  • CAS 采用 “乐观重试” 策略：不认为竞争会一直发生，所以不会让协程 “等”，而是让它 “再试一次”；
  • 重试过程中，协程始终处于 “运行态”（只是执行循环和 CAS 指令），不需要操作系统保存 / 恢复其状态 —— 完全没有上下文切换的开销。

3. 无锁调度开销，逻辑极简

  • CAS 不需要维护 “等待队列”，也不需要操作系统参与调度：每个协程都是独立重试，直到 CAS 成功；
  • 即使竞争激烈，最多只是多重试几次（每次重试就是一条 CAS 指令，几纳秒），远比 “阻塞→唤醒” 的开销小。

三、直观对比：锁 vs CAS 的并发处理流程（以 2 个协程竞争为例）

流程步骤 互斥锁（有锁竞争） CAS（无锁重试）
1. 初始状态 协程 A、B 都要修改变量 x 协程 A、B 都要修改变量 x
2. 竞争发生 协程 A 拿到锁，协程 B 拿锁失败 协程 A、B 同时读取 x 的旧值（原子读取）
3. 竞争处理 协程 B 被操作系统挂起（内核态切换 + 上下文保存），进入等待队列 协程 A CAS 成功（x 被修改），协程 B CAS 失败
4. 后续操作 协程 A 释放锁 → 操作系统唤醒协程 B（内核态切换 + 上下文恢复）→ 协程 B 拿锁执行 协程 B 循环重试：重新读 x 的新值 → 计算新值 → CAS 成功
核心开销 内核态切换、上下文切换、队列调度 少量用户态重试（CAS 指令）

四、CAS 不是 “完全无开销”，只是 “无锁竞争的开销”

要注意：CAS 不是没有任何开销，只是避免了 “锁竞争的核心开销”（内核切换、上下文切换）：

  • CAS 的开销主要来自 “重试”：如果并发竞争极其激烈（比如上百个协程同时修改一个变量），可能需要重试很多次才能成功，这会消耗 CPU 资源；
  • 但即便如此，重试的开销（每次几纳秒）也远小于锁竞争的开销（每次几十到几百纳秒）—— 只有在竞争极端激烈时，两者性能才会接近（但这种场景本身就需要优化，而非单纯依赖锁或 CAS）。

五、总结：CAS 避免锁竞争开销的本质

CAS 之所以能避免锁竞争的开销，核心是 用 “用户态乐观重试” 替代了 “内核态阻塞等待”：

  1. 无内核态切换：全程在用户态执行原子指令，不依赖操作系统内核；
  2. 无阻塞 / 唤醒：协程始终运行，不需要保存 / 恢复上下文；
  3. 无调度开销：不需要维护等待队列，逻辑极简。

补充3：M-P-G 模型

一、M、P、G 三者的定义与核心作用

M-P-G 是 Go 调度器实现「高并发协程调度」的核心模型，三者各司其职，协同完成 Goroutine 的生命周期管理：

组件 英文全称 中文含义 核心定义 核心作用
G Goroutine 协程 用户态轻量级线程，Go 程序的并发执行单元（如 go func() 启动的实例） 承载业务逻辑（函数代码），拥有独立的用户态栈（初始 2KB，动态扩容），无直接 CPU 执行权，需绑定到 M 才能运行
M Machine 内核线程 操作系统内核管理的物理线程（与 OS Thread 一一对应） 执行 Goroutine 的 “载体”，拥有内核态栈和 CPU 执行权；一个 M 同一时间只能绑定一个 P，且只能运行一个 G
P Logical Processor 逻辑处理器 连接 M 和 G 的 “中间桥梁”，持有 Goroutine 就绪队列和调度资源 1. 维护本地 Goroutine 就绪队列（LRQ），存储待执行的 G；2. 持有调度所需的资源（如内存分配缓存、GOMAXPROCS 配额）；3. 实现 M 和 G 的解耦，让 P 成为 “可调度单元”

通俗类比：

  • G 是 “打工人”（有任务要做，但没工具和场地）；
  • M 是 “干活的机器”（有 CPU 执行权，但需要场地才能开工）；
  • P 是 “工作场地”（有工具和待办任务列表，机器必须绑定场地才能让工人干活）；
  • GOMAXPROCS 是 “场地总数”（默认等于 CPU 核心数），决定了 Go 程序同时能 “开工” 的最大场地数（即最大并发执行的 M 数）。

二、P 的核心作用（调度器的 “中枢”）

P 是 M-P-G 模型的核心，没有 P 就无法实现 Goroutine 的高效调度，其作用可概括为 3 点：

  1. 资源持有与配额控制：P 持有调度所需的核心资源（如内存分配的 mcache、栈扩容的相关参数），只有绑定了 P 的 M 才能访问这些资源，确保资源隔离和高效利用；同时 P 的数量由 GOMAXPROCS 限制（默认等于 CPU 核心数），避免创建过多 M 导致操作系统线程上下文切换泛滥。
  2. 本地就绪队列（LRQ）管理：每个 P 维护一个「本地 Goroutine 就绪队列」（非全局队列），Goroutine 创建后（如 go func()）会优先放入当前 P 的 LRQ；这样设计是为了减少全局队列的锁竞争（全局队列 GQ 仅用于 P 之间的负载均衡），提升调度效率。
  3. M 与 G 的调度中介：
    • 当 P 的 LRQ 有就绪 G 时，绑定的 M 会从 LRQ 取出 G 执行（G -> M 绑定）；
    • 当 G 执行阻塞 / 结束时，M 会从 P 的 LRQ 取下一个 G 继续执行，实现 “M 复用”（避免频繁创建销毁 M）；
    • 当 P 的 LRQ 为空时，会触发「工作窃取（Work Stealing）」：从其他 P 的 LRQ 或全局队列 GQ 偷取 G，确保 P 的资源不闲置。

三、当 Goroutine 执行阻塞式系统调用时，M 与 P 的绑定关系处理

这是面试的核心考点 ——Go 调度器通过「P 转移」机制，避免 P 资源闲置，确保并发效率不降低。具体处理流程分 3 步：

1. 前提：明确 “阻塞式系统调用” 的场景

指会导致 M（内核线程）被操作系统挂起的系统调用，如：

  • 时间睡眠（time.Sleep(1s) 底层调用 nanosleep）；
  • 文件 I/O（os.Read/os.Write，未使用 runtime 封装的非阻塞接口）；
  • 锁竞争阻塞（如 sync.Mutex 竞争失败，底层调用 futex 系统调用）；
  • 其他内核态阻塞操作（如 net.Dial 未使用 netpoller 机制）。

2. 调度器的处理流程（核心是 “P 不闲置”）

假设当前状态：G1 绑定 M1，M1 绑定 P1，P1 的 LRQ 还有 G2、G3 待执行。当 G1 执行阻塞式系统调用时：

  1. M1 与 P1 解绑：Go runtime 检测到 G1 触发阻塞式系统调用后，会主动将 M1 与 P1 解除绑定（P1 释放 M1 的持有权）；此时 M1 会带着 G1 进入「阻塞状态」（被操作系统挂起），等待系统调用结果。
  2. P1 重新绑定新的 M2：P1 释放 M1 后，会立即从「空闲 M 池」中取一个空闲的 M2（若没有则创建新的 M2），并与 M2 绑定；绑定后，M2 会从 P1 的 LRQ 中取出下一个就绪 G（如 G2），继续执行 ——P1 的资源没有闲置，并发效率不受影响。
  3. 阻塞结束后 M1 的处理：当 G1 的阻塞式系统调用完成（如 time.Sleep 结束），G1 会被标记为「就绪状态」：
    • 若此时 P1 仍有空闲（或 M2 已执行完 G2、G3），G1 会被放入 P1 的 LRQ，等待 M2 执行；
    • 若 P1 已被其他 M 占用且 LRQ 满，G1 会被放入「全局就绪队列 GQ」，等待其他 P 通过工作窃取获取；
    • M1 会被放回「空闲 M 池」，等待下次被 P 绑定使用（M 复用，减少内核线程创建开销）。

3.为什么G1 不找 M1，而是等 M2？

在 G1 执行阻塞式系统调用时，调度器已经做了关键操作：

  1. G1 触发阻塞（如 time.Sleep），导致其绑定的 M1 被操作系统挂起（内核态阻塞）；
  2. Go 调度器为了不浪费 P1 的资源（P1 持有 G2、G3 等就绪协程），会主动解除 M1 与 P1 的绑定关系；M1 会立即被放回空闲池（先解绑入池，再挂起，此时 M1 仍处于操作系统挂起状态，但已不属于任何 P，进入 “待命” 模式）
  3. P1 立即从「空闲 M 池」找了新的内核线程 M2 绑定，M2 开始执行 P1 本地队列（LRQ）里的 G2、G3。

此时的状态是：

  • M1：挂起中（带着 G1 阻塞），与 P1 完全解绑，不属于任何 P；
  • M2：活跃中，与 P1 绑定，正在执行 P1 的 G；
  • P1：正常工作，LRQ 里有就绪 G，绑定的执行载体是 M2；
  • G1：阻塞中，属于 “无执行载体” 的状态。

G -> M的绑定时机

所有 G -> M 绑定的核心前提：M 已与某个 P 绑定（M 必须通过 P 获得调度资源，才能执行 G），绑定过程由 Go runtime 自动完成，开发者无感知。

场景 1：G 被创建后首次执行（最常见）

  • 时机：当通过 go func() 创建新 G 时，G 会被放入当前 P 的 LRQ（本地就绪队列）；当 M（与该 P 绑定）处理完当前 G 后，从 LRQ 取出下一个 G 时，会触发 G -> M 绑定。
  • 流程：go func() 创建 G → G 进入 P 的 LRQ → M 从 LRQ 取 G → 绑定 G 到 M → M 执行 G 的代码。

场景 2：G 阻塞后恢复就绪（如系统调用、通道阻塞结束）

  • 时机：G 因阻塞（如 time.Sleep、<-chan）被从 M 上解绑后，阻塞结束会被放回 P 的 LRQ 或全局 GQ；当 M（与 P 绑定）从队列中取出这个 G 时，重新触发 G -> M 绑定。
  • 例子：之前的 G1 阻塞结束后放入 P1 的 LRQ，M2（与 P1 绑定）取 G1 时，G1 与 M2 绑定并执行。

场景 3：P 工作窃取（Work Stealing）获得 G

  • 时机：当 P 的 LRQ 为空时，会从其他 P 的 LRQ 或全局 GQ 偷取 G；偷取成功后，M（与当前 P 绑定）会取出偷来的 G，触发 G -> M 绑定。
  • 流程：P2 的 LRQ 空 → 从 P1 的 LRQ 偷取 G3 → M3（与 P2 绑定）取 G3 → 绑定 G3 到 M3 并执行。

场景 4：G 被抢占后重新调度

  • 时机：Go 1.14+ 支持抢占式调度，若 G 持续运行超过 10ms，runtime 会强制将其从 M 上解绑，放入 LRQ；当 M 再次从 LRQ 取出这个 G 时，重新绑定 G -> M。

场景 5：主 Goroutine（main）的绑定

  • 时机：Go 程序启动时，runtime 会创建主 Goroutine（执行 main 函数），并自动将其与初始 M（程序启动时创建的第一个 M）绑定 —— 这是整个程序的第一个 G -> M 绑定。

核心原因：G1 依赖 P1 的调度资源，而非依赖 M1

Goroutine（G）的调度权由 P（逻辑处理器） 掌控，而非 M（内核线程）—— M 只是 “干活的工具”，P 才是 “分配任务的管理者”。具体来说：

  1. G1 属于 P1 的 “任务池”：G1 最初是被分配到 P1 的 LRQ 中的，其调度依赖 P1 的资源（如内存分配缓存、调度权限）。即使 M1 是之前执行 G1 的载体，但 M1 已与 P1 解绑，不再拥有访问 P1 资源的权限；
  2. M2 是 P1 当前的 “合法工具”：P1 现在绑定的是 M2，M2 正在持续从 P1 的 LRQ 中取任务执行。G1 阻塞结束后，要回到 P1 的 LRQ 排队，自然要等 P1 当前的执行载体（M2）来处理；
  3. M1 已 “失业”，无法直接执行 G1：M1 被放回「空闲 M 池」后，处于 “无 P 绑定” 的状态 —— 没有 P 分配任务，M1 即使被唤醒，也没有可执行的 G（G 都在 P 的 LRQ 或全局 GQ 中）。它必须等待某个 P 解绑后，重新绑定 P 才能获得任务。

举个通俗例子

  • P1 = 一个车间（有任务列表、工具）；
  • M1 = 车间最初的工人 A；
  • M2 = 车间后来的工人 B；
  • G1 = 车间的任务 1，G2/G3 = 车间的任务 2/3。

流程：

  1. 工人 A（M1）带着任务 1（G1）去处理，结果任务 1 卡住了（阻塞）；
  2. 车间主任（调度器）为了不浪费车间（P1）的产能，让工人 A（M1）先去休息（挂起），同时找了工人 B（M2）来车间干活（M2 绑定 P1）；
  3. 工人 B（M2）开始处理车间里剩下的任务 2、3（G2/G3）；
  4. 后来任务 1（G1）不卡了（阻塞结束），回到车间排队（P1 的 LRQ）—— 此时车间里只有工人 B（M2）在干活，所以任务 1 只能等工人 B 处理完手头的任务，再轮到自己；
  5. 工人 A（M1）休息完后，回到 “工人待命区”（空闲 M 池），等着其他车间（其他 P）需要人时再去干活（重新绑定 P）。

这里的关键是：任务属于车间（P），而非某个工人（M） —— 工人只是车间的 “临时执行者”，车间换了工人，任务排队只能等当前车间的工人处理。

关键结论：

阻塞式系统调用只会导致「M 被挂起」，但 P 会快速绑定新的 M 继续执行其他 G，不会因为一个 G 阻塞而导致整个 P 的资源闲置 —— 这是 Go 能高效支持大量阻塞协程的核心原因。

四、非阻塞式系统调用（如网络 I/O）的特殊处理

补充一个高频延伸点：当 G 执行「非阻塞式系统调用」（如 net.Conn.Read，底层通过 netpoller 机制实现）时，调度器的处理逻辑不同：

  • 此时 G 会被标记为「IO 阻塞状态」，从 M 上解绑，放入 runtime 的「IO 等待队列」；
  • M 不会被挂起，继续从 P 的 LRQ 取其他 G 执行；
  • 当 IO 事件就绪（如数据到达），netpoller 会将 G 从 IO 等待队列移回 P 的 LRQ，等待 M 执行；
  • 这种场景下，M 与 P 不会解绑，P 资源利用率更高（无需重新绑定新 M）。

五、总结

  1. M-P-G 定义：
    • G：协程（业务逻辑载体）；M：内核线程（执行载体）；P：逻辑处理器（调度中枢，连接 M 和 G）。
  2. P 的核心作用：
    • 持有调度资源和本地就绪队列（LRQ）；
    • 限制最大并发数（GOMAXPROCS）；
    • 实现 M 与 G 的解耦，支持工作窃取和 M 复用。
  3. 阻塞式系统调用时的 M-P 绑定处理：
    • G 阻塞导致 M 被挂起，M 与 P 立即解绑；
    • P 快速绑定新的空闲 M，继续执行 LRQ 中的其他 G；
    • G 阻塞结束后，被放回就绪队列（LRQ/GQ），M 被复用 —— 确保 P 资源不闲置，并发效率不受影响。

补充4：什么是Go runtime

Go runtime（Go 运行时）是 Go 语言内置的 “底层支撑系统”—— 它不是用户代码的一部分，而是随 Go 程序一起编译、打包、运行的核心组件集合，本质是 “介于 Go 应用代码和操作系统内核之间的中间层”，负责管理 Go 程序的并发、内存、调度、垃圾回收等关键能力，让开发者无需关注底层细节，就能写出高效的并发程序。

简单说：Go 开发者写的代码（如 go func()、chan、sync 包）是 “业务逻辑”，而 runtime 是 “底层引擎”—— 引擎负责把业务逻辑高效地映射到操作系统上执行，这也是 Go 语言 “简单易用又高性能” 的核心原因。

Go runtime 的核心功能

runtime 隐藏了大量底层细节，核心功能围绕 “让 Go 程序高效运行” 展开，尤其是并发相关能力：

1. 协程（Goroutine）生命周期管理

  • 负责 Goroutine 的创建（go func() 时，runtime 会分配 Goroutine 栈、程序计数器等资源）、调度（M-P-G 模型的实现者）、销毁（Goroutine 执行完后，回收栈内存等资源）；
  • 提供 Goroutine 切换的底层支持（用户态轻量切换，无需内核介入），以及抢占式调度（Go 1.14+），避免某个 Goroutine 长期占用 CPU。

2. M-P-G 调度器实现

  • runtime 是 M-P-G 模型的 “执行者”：管理 M（内核线程）、P（逻辑处理器）、G（协程）的生命周期和绑定关系；
  • 处理 Goroutine 阻塞场景（如系统调用、通道阻塞、锁竞争）：解绑 M-P、复用 M、工作窃取（Work Stealing）等逻辑，都是 runtime 内部实现的。

3. 内存管理与垃圾回收（GC）

  • 内存分配：提供高效的内存分配机制（如 TCmalloc 风格的分级分配、线程缓存 MCache、中心缓存 MCentral、页堆 MHeap），避免频繁调用系统调用分配内存；
  • 垃圾回收：自动回收不再使用的内存（如未被引用的变量、切片、结构体），Go 1.5+ 后的并发标记清除（CMS）+ 三色标记法，让 GC 停顿时间极短（毫秒级甚至微秒级），不影响业务运行。

4. 系统调用封装与适配

  • 对操作系统的底层系统调用（如文件 I/O、网络 I/O、时间睡眠）进行封装，提供统一的 Go 标准库接口（如 os.Read、net.Conn.Read、time.Sleep）；
  • 优化网络 I/O 性能：通过 netpoller 机制（基于操作系统的 I/O 多路复用，如 epoll、kqueue），将网络 I/O 从 “阻塞式系统调用” 转为 “非阻塞式”，避免 Goroutine 频繁阻塞导致的 M-P 解绑开销。

5. 并发原语的底层支持

  • Go 标准库中 sync 包（Mutex、RWMutex、WaitGroup、Cond）、sync/atomic 包的原子操作，其底层实现都依赖 runtime；
  • 例如：Mutex 的自旋锁 + 信号量机制、WaitGroup 的计数器管理、原子操作的 CPU 指令封装，都是 runtime 提供的底层能力。

6. 其他核心支撑

  • 栈管理：Goroutine 的栈是动态扩容 / 缩容的（初始 2KB，最大可到 GB 级），由 runtime 自动管理，避免栈溢出或内存浪费；
  • 类型信息与反射：runtime 存储了 Go 程序的类型元数据，支持 reflect 包的反射功能；
  • 程序启动与退出：Go 程序启动时，runtime 会先初始化（如设置 GOMAXPROCS、创建初始 M 和 P、启动 GC 协程），再执行 main 函数；程序退出时，runtime 会回收资源、等待所有 Goroutine 完成（若用 WaitGroup）。

举个例子：runtime 如何工作？（结合之前的 M-P-G 模型）

runtime 会做以下关键操作：

  1. 程序启动：runtime 初始化，设置 GOMAXPROCS（默认 = CPU 核心数），创建初始的 M（内核线程）和 P（逻辑处理器）；
  2. 执行 main 函数：main 函数本身也是一个 Goroutine（主 Goroutine）；
  3. 创建协程：执行 go func() 时，runtime 会创建一个新的 Goroutine G1，分配栈空间，将其放入 P 的本地就绪队列（LRQ）；
  4. 调度执行：M 绑定 P 后，从 LRQ 中取出 G1，切换到 G1 执行（用户态协程切换），打印 “hello goroutine”；
  5. 程序退出：G1 执行完后，runtime 回收其资源；主 Goroutine 执行完后，runtime 等待所有 Goroutine 完成（若有），然后退出。
