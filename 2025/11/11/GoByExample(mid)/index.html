1.协程

3.核心语法规律

  • 前半部分 func(...) { ... } 是「匿名函数定义」；
  • 后半部分 (...) 是「函数调用传参」，必须紧跟在函数体后面；

理解：无缓冲通道的非阻塞发送

要彻底理解「无缓冲通道的非阻塞发送」，核心是抓住 无缓冲通道的本质：发送和接收必须 “手递手” 同步完成—— 发送方会阻塞到有接收方就绪，接收方也会阻塞到有发送方就绪。而非阻塞发送的关键是：尝试 “手递手”，如果找不到接收方，就立刻放弃，走 default 分支。

9.通道的关闭

注：为什么把 close(jobs) 注释掉后，会运行时死锁（deadlock）

核心原因：子协程永远不知道 “任务已经发完了”

要理解这个问题，必须回到「发送方和接收方的协作逻辑」：

1. 注释 close(jobs) 后，各方的状态

  • 主协程：发送完 3 个任务（1、2、3）后，打印 sent all jobs，然后阻塞在 <-done，等待子协程发送 “处理完毕” 的信号；
  • 子协程：在for循环中持续执行j, more := <-jobs：
    • 首先接收缓冲区中的 3 个任务（1、2、3），more 始终为 true（因为通道没关闭）；
    • 处理完这 3 个任务后，缓冲区空了，但通道仍处于「未关闭状态」；
    • 子协程会继续阻塞在 <-jobs，等待新的任务（它不知道主协程已经不发了）；
  • 关键矛盾：主协程在等 done 信号，子协程在等新任务，双方互相等待，永远无法推进 —— 这就是死锁。

2. close(jobs) 的核心作用：打破 “互相等待”

close(jobs) 不是 “可选操作”，而是「发送方向接收方传递 “任务终止” 的唯一信号」：

  • 关闭通道后，子协程处理完缓冲区的 3 个任务，再执行 j, more := <-jobs 时，会得到 more = false（通道关闭 + 缓冲区空）；
  • 此时子协程才知道 “没有更多任务了”，会执行 done <- true，通知主协程退出；
  • 主协程收到 done 信号后，解除阻塞，程序正常结束。

11.Timer

Timer 的核心基础

1. Timer 的本质

Timer 是一个 “单次定时任务器”，核心逻辑：

  • 调用 time.NewTimer(d) 时，启动一个后台协程；
  • 延迟 d 时间后，后台协程向 Timer.C（一个 chan time.Time 通道）发送当前时间值；
  • 使用者通过「接收 Timer.C 的信号」，感知定时时间到达；
  • 核心：Timer 触发一次后就失效（单次触发），如需重复定时需重新创建或重置。

2. 核心属性与方法

属性 / 方法 作用说明
Timer.C 只读通道（<-chan time.Time），定时时间到后会收到一个时间信号（信号本身可忽略）；
time.NewTimer(d) 创建 Timer 实例，延迟 d 后触发；
timer.Stop() 提前停止 Timer：返回 true 表示 Timer 未触发且成功停止，false 表示已触发或已停止；
timer.Reset(d) 重置 Timer：停止当前未触发的 Timer，重新设置延迟 d（可实现重复定时）；

3. 与 time.After 的区别（关键）

  • time.After(d)：是 Timer 的简化版，返回 Timer.C 通道，无法提前停止（Timer 会在触发后自动回收）；
  • Timer：支持 Stop() 提前停止，更灵活（适合需要取消定时的场景）；
  • 核心差异：Timer 可控制（停止 / 重置），time.After 不可控制。

注：关键特性

1. Timer 触发后，Timer.C 通道不会自动关闭

Timer 触发一次后，Timer.C 通道会保留发送的时间值（缓冲区容量 1），再次接收会立即返回该值，不会阻塞：

  • 运行结果：第二次接收成功（无阻塞）；
  • 原因：Timer.C 是容量为 1 的缓冲通道，触发后数据会保留在通道中。

2. 提前停止 Timer 后，Timer.C 通道会永久阻塞

如案例 2 所示，timer.Stop() 后，后台协程不会向 Timer.C 发送信号，通道会一直处于 “无数据” 状态，接收操作会永久阻塞 —— 若子协程中只有 <-timer.C 而无其他逻辑，会导致子协程泄漏（一直占用资源）。

12.Ticker

Ticker 的核心基础

1. Ticker 的本质

Ticker 是 “周期性定时任务器”，核心逻辑：

  • 调用 time.NewTicker(d) 时，启动一个后台协程；
  • 每隔 d 时间，后台协程向 Ticker.C（chan time.Time 通道）发送当前时间值；
  • 使用者通过接收 Ticker.C 的信号，触发周期性任务；
  • 核心：只要不停止 Ticker，就会一直周期性触发（直到程序退出或主动停止）。

13.工作池

工作池的核心基础

工作池的本质是「生产者 - 消费者模型」的具体化，三个核心组件缺一不可：

  1. 任务生产者：负责生成任务，发送到「任务通道（Jobs）」（代码中主协程是生产者）；
  2. 工作协程（Worker）：提前创建的固定数量协程，从任务通道获取任务执行，执行结果发送到「结果通道（Results）」（代码中 worker 函数是 Worker）；
  3. 任务 / 结果通道：
    • 任务通道（jobs <-chan int）：单向接收通道，Worker 只能从这里拿任务；
    • 结果通道（results chan<- int）：单向发送通道，Worker 只能往这里存结果；
    • 通道的缓冲容量：通常设置为任务总数（避免生产者发送任务时阻塞）。

核心逻辑：生产者往 Jobs 发任务 → 多个 Worker 并发抢任务执行 → 执行结果存入 Results → 生产者等待所有结果处理完毕。

关键特性

1. 工作池的核心优势：并发控制 + 资源复用

  • 并发控制：通过固定 Worker 数量（3 个），限制最大并发数，避免系统因协程过多导致 CPU / 内存过载；
  • 资源复用：Worker 是提前创建的，处理完一个任务后会继续处理下一个，无需频繁创建销毁协程（创建协程有资源开销，频繁创建会降低效率）。

2. 为什么用 for range 遍历 jobs 通道？

for range jobs 会自动处理「任务接收 + 通道关闭检测」：

  • 有任务时，接收任务并执行；
  • 通道关闭且无任务时，自动退出循环，Worker 协程自然结束（无需手动判断 ok 值）；

3. 缓冲通道的作用：解耦生产者和消费者

  • 任务通道（容量 5）：主协程可以一次性发送所有任务，无需等待 Worker 接收（缓冲区暂存任务）；
  • 结果通道（容量 5）：Worker 处理完任务后可以立即发送结果，无需等待主协程接收（缓冲区暂存结果）；
  • 如果用无缓冲通道，主协程发送任务时会阻塞，直到有 Worker 接收，效率会降低（缓冲通道起到 “削峰填谷” 的作用）。

4. 扩展：动态控制任务数或 Worker 数

  • 动态任务数：如果任务数不固定（比如从网络接收任务），可以不关闭 jobs 通道，持续往里面发送任务，直到所有任务发送完毕再关闭；
  • 动态 Worker 数：可以根据系统负载动态调整 Worker 数量（比如通过 sync.WaitGroup 管理 Worker 生命周期），但实际开发中更常用固定 Worker 数（简单可控）。

5. 避免 Worker 泄漏

如果忘记关闭 jobs 通道，且任务数有限，Worker 会永久阻塞在 for range jobs 中（因为通道未关闭，会一直等待新任务），导致协程泄漏。解决办法：

  • 所有任务发送完毕后，必须关闭 jobs 通道；
  • 或用 context.Context 通知 Worker 退出（适合动态停止场景）。

14.WaitGroup

WaitGroup 是 Go 中最常用的协程同步工具之一（尤其适合 “无返回结果的批量任务”，如日志打印、资源清理），比之前用「通道（done chan bool）」同步更简洁（无需手动创建通道、发送信号）。

WaitGroup 的核心基础

WaitGroup 本质是一个 “计数器”，通过三个方法协同工作，实现协程同步：

方法名 作用说明
wg.Add(n int) 增加等待的协程数量（n 是要新增的协程数），必须在启动协程前调用；
wg.Done() 减少等待的协程数量（等价于 wg.Add(-1)），表示 “当前协程执行完毕”，必须在协程退出前调用；
wg.Wait() 主协程阻塞，直到等待的协程数量变为 0（所有子协程都调用了 Done()），才会解除阻塞；

核心逻辑：

  1. 主协程用 wg.Add(n) 声明 “要等待 n 个协程”；
  2. 每个子协程启动后，用 defer wg.Done() 确保 “执行完毕后自动通知 WaitGroup”；
  3. 主协程用 wg.Wait() 阻塞，直到所有子协程都通知 “完成”，再继续执行。

关键特性

1. 必须避免的 3 个常见错误

（1）wg.Add(n) 调用时机错误（在协程内调用）

  • 后果：主协程可能在所有 wg.Add(1) 执行前就调用 wg.Wait()，此时计数为 0，主协程直接退出，子协程被终止；
  • 原则：wg.Add(n) 必须在 启动协程前 调用，确保主协程 wg.Wait() 时，计数已经正确设置。

（2）忘记调用 wg.Done() 或 defer 位置错误

  • 后果：主协程会永久阻塞在 wg.Wait()（计数永远不为 0）；
  • 原则：wg.Done() 必须用 defer 声明，且放在 协程函数开头（确保任务执行前就注册 “完成通知”，无论任务是否 panic 都能触发）。

（3）拷贝 WaitGroup 变量（值类型传递）

  • 原因：sync.WaitGroup 是值类型，拷贝后会生成新的实例，子协程操作的是拷贝后的实例，原主协程的 wg 计数不会变化；

  • 解决：传递 *sync.WaitGroup 指针（确保所有协程操作的是同一个 WaitGroup 实例）：

2. WaitGroup 与通道（chan）的选择场景

WaitGroup 和通道都能实现协程同步，但适用场景不同：

场景类型 更适合 WaitGroup 更适合通道（chan）
核心需求 等待一组协程 “全部完成”（无返回结果） 等待协程 “完成通知”+ 传递结果（或需要超时、非阻塞）

15.速率限制

速率限制是并发编程中常见的需求（如接口限流、爬虫频率控制、消息队列消费速率限制），Go 通过「通道的阻塞特性」实现速率限制，无需额外依赖第三方库，语法简洁且高效。

速率限制的核心原理

速率限制的本质是「控制 “任务执行” 与 “时间” 的匹配关系」，Go 实现的核心逻辑：

  1. 周期性信号源：用 time.Tick(d) 生成周期性信号（如每 200ms 一个信号），time.Tick(d) 本质是一个 <-chan time.Time 通道，每隔 d 时间发送一次当前时间；
  2. 信号与任务绑定：每个任务执行前，必须先接收一个周期性信号 —— 通过通道的阻塞特性，强制任务执行频率不超过信号生成频率（如每 200ms 只能接收一个信号，所以每秒最多执行 5 个任务）；
  3. 突发流量支持：用缓冲通道存储 “预分配的信号”，允许短时间内处理多个突发任务（缓冲容量 = 最大突发数），后续任务仍按固定速率执行。

核心思想：用 “信号发放速率” 控制 “任务执行速率”，通道的阻塞特性天然实现了 “任务等待信号” 的逻辑，无需手动计时或加锁。

16.原子计数器

原子操作的底层是通过 CPU 硬件指令 实现的（如 CAS 指令），比互斥锁（Mutex）更高效（无锁竞争、上下文切换开销）。

原子计数器的核心基础

1.核心问题：为什么需要原子计数器？

多协程并发修改同一个变量时，会出现「数据竞争」—— 普通变量的增减操作（如 ops++）不是原子的，本质是 “读取→修改→写入” 三步，多协程同时操作可能导致数据错乱。

示例（普通变量的并发问题）：

  • 问题：ops++ 被拆解为「读 ops 值 → 加 1 → 写回 ops」，多个协程可能同时读同一个值，导致 “加 1 操作被覆盖”（如协程 A 和 B 都读 ops=100，加 1 后都写回 101，实际应是 102）；
  • 解决：用 sync/atomic 包的原子操作，将 “读 - 改 - 写” 封装为一个不可中断的硬件级原子指令，避免数据竞争。

2. sync/atomic 核心原子计数方法（针对无符号整数）

方法名 作用说明
atomic.AddUint64(&x, n) 对无符号 64 位整数 x 原子加 n，返回加后的新值（x 必须传指针）；
atomic.LoadUint64(&x) 原子读取 x 的值（避免读取时被其他协程的写操作干扰）；
atomic.StoreUint64(&x, v) 原子将 v 赋值给 x（避免赋值时被其他协程的读 / 写操作干扰）；
atomic.CompareAndSwapUint64(&x, old, new) CAS 操作：若 x 等于 old，则原子替换为 new，返回是否成功（乐观锁核心）；
  • 注意：sync/atomic 针对不同类型提供了对应的方法（如 AddInt32/AddInt64 对应有符号整数），需根据变量类型选择，不可混用。

3. 原子操作的核心优势（对比 sync.Mutex）

特性 原子操作（atomic） 互斥锁（sync.Mutex）
底层实现 CPU 硬件原子指令（如 CAS），无锁机制 操作系统互斥锁，有锁竞争、上下文切换开销
性能 极高（单次操作纳秒级），适合高频次操作 较低（单次操作微秒级），适合复杂临界区
适用场景 简单的数值增减、读取（如计数器、统计量） 复杂逻辑（如多步操作、多变量修改）
代码复杂度 简单（一行调用） 稍复杂（需手动加锁 Lock()/ 解锁 Unlock()）

注：关键特性

1. 必须避免的 3 个常见错误

（1）原子操作变量未传指针

  • 原因：原子操作需要直接修改变量的内存地址，必须传指针（&ops），否则无法保证操作的原子性；
  • 原则：所有 atomic 包的方法，第一个参数都是变量的指针（如 *uint64、*int32）。

（2）混用原子操作和普通操作

  • 后果：普通操作（ops++）非原子，会干扰原子操作的安全性，导致数据错乱；
  • 原则：一个变量若用 atomic 包操作，就必须全程用原子操作（增、删、读、写），不能混用普通操作。

（3）原子操作用于复杂逻辑

  • 问题：atomic.AddUint64 是原子的，但 “加 1 + 判断” 整体不是原子的 —— 可能多个协程同时加 1 后，都判断超过阈值，导致重复执行后续逻辑；
  • 解决：复杂逻辑需用 sync.Mutex 加锁，确保整个临界区的原子性：

2. 原子操作的适用场景（vs 互斥锁）

  • 优先用原子操作的场景：
    1. 简单数值计数（如接口调用次数、任务执行量、错误统计）；
    2. 高频次操作（如每秒上万次的计数，原子操作性能优势明显）；
    3. 单一变量的读 / 写 / 增减（原子操作仅支持单一变量）。
  • 优先用互斥锁的场景：
    1. 复杂逻辑（如多步操作、多变量修改）；
    2. 临界区代码较长（原子操作仅适合 “短操作”）；
    3. 需要条件变量（如 sync.Cond）配合的场景。

（4）结构体拷贝导致锁失效

  • 原因：sync.Mutex 是值类型，拷贝后会生成新的锁实例 —— 两个协程操作的是不同的锁，无法保护同一个 counters（实际 c2 的 counters 也是拷贝的，并非同一个 map）；
  • 解决：传递 *Container 指针（避免值拷贝），确保所有协程操作的是同一个锁和同一个共享资源。

2. 扩展：读写互斥锁（sync.RWMutex）

如果场景是「读多写少」（如频繁读取 counters，偶尔修改），可以用 sync.RWMutex 替代 sync.Mutex，提升并发读性能：

  • RWMutex 支持两种锁：读锁（RLock()/RUnlock()）和写锁（Lock()/Unlock()）；
  • 多个协程可同时获取读锁（并发读），但写锁会阻塞所有读锁和其他写锁（互斥写）；

18.状态协程

状态协程的核心优势是「天然并发安全，无需手动加锁」—— 所有对共享状态的操作都集中在一个协程中，不存在多协程并发修改的问题，通道的 FIFO 特性还能保证请求处理的顺序性。

核心原理：状态协程的 “单线程处理” 思想

传统多协程操作共享状态（如 map），需要用锁（sync.Mutex）保护临界区；而状态协程的思路是：

  1. 状态隔离：共享状态（state map[int]int）只由一个 “状态协程” 持有，其他协程无法直接访问；
  2. 通道通信：读写请求通过专用通道（reads chan readOp 读请求、writes chan writeOp 写请求）发送给状态协程；
  3. 串行处理：状态协程通过 select 监听读写通道，收到请求后串行处理（同一时间只处理一个请求），天然避免数据竞争。

简单说：用 “协程串行处理” 替代 “多协程并发竞争 + 锁”，通道负责请求传递和同步，状态协程负责逻辑执行，既安全又简洁。

补充1：通道为什么天生线程安全？

Go 的通道在底层由运行时（runtime）管理，内置了同步机制，核心保障有两点：

  1. 发送 / 接收操作是原子的：
    • 发送操作（ch <- data）：将数据存入通道缓冲区（或直接传递给接收方）的整个过程，不会被其他协程打断；
    • 接收操作（data := <-ch）：从通道缓冲区取出数据（或直接接收发送方数据）的整个过程，也不会被其他协程打断；
    • 原子性意味着：不会出现 “多个协程同时写通道导致数据错乱”“多个协程同时读通道导致重复读取” 的问题。
  2. 运行时的锁机制：
    • 通道底层维护了一个互斥锁（mutex），每次发送 / 接收操作都会先获取锁，操作完成后释放锁；
    • 锁机制确保：同一时间只有一个协程能执行通道的发送或接收操作，天然避免了数据竞争。

需要注意的 “非线程安全” 场景（不是通道本身的问题）

通道本身是线程安全的，但以下场景可能出现问题，需要避免 —— 本质是 “用户代码逻辑问题”，而非通道的安全问题：

  1. 向已关闭的通道发送数据：会触发 panic（通道关闭后禁止发送，这是语法规则，不是线程安全问题）；
  2. 重复关闭通道：会触发 panic（同样是语法规则）；
  3. 通道的 “外层数据” 共享：如果多个协程通过通道传递的是 “指针 / 引用类型”（如切片、map），且同时修改该数据的内容，会出现数据竞争 —— 但这是 “共享数据的线程安全问题”，和通道本身无关。

示例（引用类型的风险）：

  • 问题：切片是引用类型，两个协程共享同一个切片的底层数据，修改时出现数据竞争；
  • 解决：传递值类型（如 int、struct），或传递引用类型时加锁（如 sync.Mutex），或传递副本（如 append 生成新切片）。

Go 通道 vs Java 多线程

Go 的通道（chan）和 Java 的多线程模型（核心是 Thread + synchronized/Lock + 共享内存），本质是 两种不同的并发编程范式：

  • Go 通道是「通过通信共享内存（CSP 模型）」的核心载体，强调 “协程间通过通道传递数据，而非直接共享内存”；
  • Java 多线程是「通过共享内存通信」的经典模型，强调 “多线程竞争共享资源，需手动加锁同步”。
维度 Go 通道（基于 CSP 模型） Java 多线程（基于共享内存模型）
并发单元 协程（Goroutine）：轻量级（内存占用～2KB，调度成本低），支持百万级并发 线程（Thread）：重量级（内存占用～1MB，调度成本高），支持千级并发（受系统线程数限制）
通信 / 同步方式 通道传递数据（chan），天然支持同步（阻塞发送 / 接收）、缓冲、关闭通知 共享内存（如静态变量、对象字段），需手动用 synchronized/Lock 加锁，Condition/CountDownLatch 同步
线程安全保障 通道本身线程安全（底层内置锁 + 原子操作），无需额外加锁 共享资源需手动保障线程安全（加锁 / 解锁），易出现死锁、数据竞争
编程思维 关注 “数据如何在协程间流动”，逻辑更线性、简洁 关注 “如何保护共享资源不被并发修改”，需处理锁竞争、同步细节

补充2：CAS 为什么能避免锁竞争的开销

要搞懂 “CAS 为什么能避免锁竞争的开销”，核心是先明确 “锁竞争的开销到底是什么”，再对比 CAS 的 “无锁机制” 如何绕开这些开销 —— 本质是「锁用 “阻塞等待” 解决并发，CAS 用 “循环重试” 解决并发」，后者省去了阻塞 / 唤醒的核心开销。

一、先搞懂：锁竞争的开销到底来自哪里？

我们以之前提到的 sync.Mutex（互斥锁）为例，当多个协程竞争同一个锁时，开销主要集中在 3 个核心环节（这些都是 CAS 没有的）：

1. 内核态与用户态切换（最大开销）

  • 互斥锁的实现依赖 操作系统内核：当协程 A 拿到锁后，其他协程（B、C、D）尝试加锁会失败；
  • 失败的协程不会一直占用 CPU 重试，而是会被操作系统 挂起（从 “运行态” 转为 “阻塞态”），这个过程需要从 “用户态” 切换到 “内核态”（操作系统负责调度）；
  • 当协程 A 释放锁后，操作系统还要从 “内核态” 切换回 “用户态”，唤醒被挂起的协程（B、C、D 中的一个），让它拿到锁继续执行；
  • 这种 “用户态↔内核态” 切换的开销极大（单次切换约几十到几百纳秒），远超 CAS 操作本身（几纳秒）。

2. 协程 / 线程上下文切换

  • 被挂起的协程，其当前的执行状态（如 CPU 寄存器值、程序计数器）需要被操作系统保存到内存中；
  • 唤醒时，又要从内存中恢复这些状态到 CPU 中 —— 这个 “保存→恢复” 的过程就是 “上下文切换”，同样有显著开销（尤其当竞争激烈、切换频繁时）。

3. 锁的调度与排队开销

  • 操作系统需要维护一个 “等待队列”，管理所有因拿不到锁而阻塞的协程；
  • 释放锁时，还要从队列中选择一个协程唤醒（公平锁 / 非公平锁的调度逻辑），这个过程也需要消耗内核资源。

简单说：锁竞争的核心开销，是 “阻塞→内核调度→唤醒→上下文切换” 这一整套流程的开销—— 本质是用 “让协程等” 的方式解决竞争，而 “等” 的过程本身就很耗资源。

二、CAS 为什么能避免这些开销？核心是 “无锁 + 用户态重试”

CAS（Compare And Swap）的核心是「不阻塞协程，仅在用户态循环重试」，完全绕开了上述锁竞争的开销，具体逻辑如下：

1. 全程在用户态执行，无内核态切换

  • CAS 是 CPU 硬件支持的原子指令（如 x86 架构的 CMPXCHG 指令），整个 “比较 + 交换” 操作都在 用户态 完成，不需要调用操作系统内核；
  • 即使 CAS 失败（遇到并发竞争），也不会挂起协程，只是在用户态重新执行 “读旧值→算新值→CAS” 的循环 —— 全程不涉及内核态切换，开销极低。

2. 无协程阻塞 / 唤醒，无上下文切换

  • CAS 采用 “乐观重试” 策略：不认为竞争会一直发生，所以不会让协程 “等”，而是让它 “再试一次”；
  • 重试过程中，协程始终处于 “运行态”（只是执行循环和 CAS 指令），不需要操作系统保存 / 恢复其状态 —— 完全没有上下文切换的开销。

3. 无锁调度开销，逻辑极简

  • CAS 不需要维护 “等待队列”，也不需要操作系统参与调度：每个协程都是独立重试，直到 CAS 成功；
  • 即使竞争激烈，最多只是多重试几次（每次重试就是一条 CAS 指令，几纳秒），远比 “阻塞→唤醒” 的开销小。

三、直观对比：锁 vs CAS 的并发处理流程（以 2 个协程竞争为例）

流程步骤 互斥锁（有锁竞争） CAS（无锁重试）
1. 初始状态 协程 A、B 都要修改变量 x 协程 A、B 都要修改变量 x
2. 竞争发生 协程 A 拿到锁，协程 B 拿锁失败 协程 A、B 同时读取 x 的旧值（原子读取）
3. 竞争处理 协程 B 被操作系统挂起（内核态切换 + 上下文保存），进入等待队列 协程 A CAS 成功（x 被修改），协程 B CAS 失败
4. 后续操作 协程 A 释放锁 → 操作系统唤醒协程 B（内核态切换 + 上下文恢复）→ 协程 B 拿锁执行 协程 B 循环重试：重新读 x 的新值 → 计算新值 → CAS 成功
核心开销 内核态切换、上下文切换、队列调度 少量用户态重试（CAS 指令）

四、CAS 不是 “完全无开销”，只是 “无锁竞争的开销”

要注意：CAS 不是没有任何开销，只是避免了 “锁竞争的核心开销”（内核切换、上下文切换）：

  • CAS 的开销主要来自 “重试”：如果并发竞争极其激烈（比如上百个协程同时修改一个变量），可能需要重试很多次才能成功，这会消耗 CPU 资源；
  • 但即便如此，重试的开销（每次几纳秒）也远小于锁竞争的开销（每次几十到几百纳秒）—— 只有在竞争极端激烈时，两者性能才会接近（但这种场景本身就需要优化，而非单纯依赖锁或 CAS）。

五、总结：CAS 避免锁竞争开销的本质

CAS 之所以能避免锁竞争的开销，核心是 用 “用户态乐观重试” 替代了 “内核态阻塞等待”：

  1. 无内核态切换：全程在用户态执行原子指令，不依赖操作系统内核；
  2. 无阻塞 / 唤醒：协程始终运行，不需要保存 / 恢复上下文；
  3. 无调度开销：不需要维护等待队列，逻辑极简。
